Цель работы

Изучить внутреннее устройство класса DbContext в Entity Framework Core, его назначение и роль при взаимодействии приложения с базой данных. Освоить механизм отслеживания изменений сущностей с помощью Change Tracker, а также реализовать полный набор CRUD-операций в приложении «Телефонная книга». В ходе выполнения работы необходимо закрепить навыки чтения, добавления, редактирования и удаления данных через ApplicationContext, внедрения контекста базы данных во ViewModel через Dependency Injection, вызова метода SaveChanges() для сохранения изменений и обновления пользовательского интерфейса после выполнения операций с базой данных.

Задание

Модернизировать приложение «Телефонная книга» для полноценной работы с базой данных с использованием Entity Framework Core. Программа должна обеспечивать: анализ структуры сгенерированного класса ApplicationContext; наличие свойства DbSet<Contact> для работы с таблицей Contacts; внедрение ApplicationContext в ContactsListViewModel и ContactEditViewModel через конструкторы; загрузку списка контактов из базы данных при запуске приложения; реализацию фильтрации контактов по имени и номеру телефона; добавление нового контакта через форму; сохранение нового контакта в базе данных с помощью метода Add() и SaveChanges(); редактирование выбранного контакта; сохранение изменённых данных в базе с помощью механизма отслеживания изменений; удаление выбранного контакта из таблицы Contacts; обновление интерфейса после добавления, редактирования и удаления данных; обработку возможных ошибок при работе с базой данных с помощью блоков try-catch; демонстрацию корректной работы всех CRUD-операций без использования дополнительных паттернов абстракции, таких как Repository.

Теоретическое обоснование

Entity Framework Core является ORM-фреймворком, который позволяет работать с базой данных через объекты C#, не выполняя ручное написание большинства SQL-запросов. Центральным элементом EF Core является класс DbContext. В проекте Lab_13 таким классом является ApplicationContext. Он связывает модель приложения с базой данных, управляет подключением, выполняет запросы, сопоставляет классы C# с таблицами базы данных и отслеживает изменения объектов.

Одной из основных частей класса DbContext является свойство DbSet<TEntity>. В данной лабораторной работе используется свойство DbSet<Contact> Contacts, которое соответствует таблице Contacts в базе данных. Через это свойство выполняются основные операции с данными: чтение, добавление, изменение и удаление контактов. Метод OnModelCreating используется для настройки соответствия между классом Contact и таблицей базы данных, а также для задания ограничений, имён столбцов и других параметров модели.

Важную роль при работе Entity Framework Core играет жизненный цикл сущностей. Когда объект загружается из базы данных через контекст, EF Core начинает отслеживать его состояние с помощью Change Tracker. У сущности может быть несколько состояний: Unchanged, Added, Modified, Deleted и Detached. Состояние Unchanged означает, что объект был загружен из базы данных и пока не изменялся. Состояние Added присваивается новому объекту, который был добавлен в DbSet, но ещё не сохранён в базе. Состояние Modified означает, что у уже существующего объекта были изменены свойства. Состояние Deleted используется для объекта, который был помечен на удаление. Состояние Detached означает, что объект не отслеживается текущим контекстом.

Жизненный цикл сущности напрямую связан с методом SaveChanges(). Само изменение свойств объекта в C# ещё не сохраняет данные в базе. До вызова SaveChanges() изменения существуют только в памяти приложения и отслеживаются контекстом. При вызове этого метода Change Tracker анализирует состояния всех объектов и формирует соответствующие SQL-команды. Для объектов в состоянии Added создаётся команда INSERT, для объектов в состоянии Modified — команда UPDATE, а для объектов в состоянии Deleted — команда DELETE. После успешного сохранения EF Core синхронизирует состояние объектов с базой данных.

Операция Create реализуется через создание нового объекта Contact и добавление его в коллекцию _context.Contacts методом Add(). После этого объект получает состояние Added, а при вызове SaveChanges() добавляется в таблицу базы данных. Операция Read выполняется с помощью LINQ-запросов к DbSet, например ToList(), Where() или FirstOrDefault(). Операция Update может выполняться без явного вызова метода Update(), если объект был получен из этого же контекста и отслеживается Change Tracker. В таком случае достаточно изменить свойства объекта и вызвать SaveChanges(). Метод Update() требуется в ситуации, когда объект не отслеживается текущим контекстом, например был создан заново или передан из другого контекста. Операция Delete выполняется с помощью метода Remove(), после чего объект получает состояние Deleted, а при вызове SaveChanges() удаляется из базы данных.

При большом количестве данных Change Tracker может влиять на производительность, так как контексту необходимо хранить информацию о загруженных объектах и проверять их изменения. Поэтому при чтении больших объёмов данных, которые не планируется изменять, можно использовать AsNoTracking(). Однако в данной лабораторной работе основной акцент сделан на понимании механизма отслеживания изменений, поэтому операции редактирования и удаления выполняются через объекты, полученные из контекста базы данных.

Описание выполненных действий

Для выполнения лабораторной работы был модернизирован проект «Телефонная книга» с названием Lab_13. На первом этапе была проанализирована структура класса ApplicationContext, созданного в предыдущей лабораторной работе. В нём было найдено свойство DbSet<Contact> Contacts, которое используется для работы с таблицей контактов. Также был рассмотрен метод OnModelCreating, где задаётся соответствие между классом Contact и таблицей Contacts.

Далее была настроена структура проекта. В проекте использовались папки Models, Data и ViewModels. В папке Models размещён класс Contact, описывающий контакт телефонной книги. В папке Data расположен класс ApplicationContext, наследуемый от DbContext. В папке ViewModels были размещены классы ContactsListViewModel, ContactEditViewModel, ObservableObject и RelayCommand. Для работы с базой данных в проект были подключены пакеты Microsoft.EntityFrameworkCore, Microsoft.EntityFrameworkCore.SqlServer, Microsoft.EntityFrameworkCore.Tools, Microsoft.EntityFrameworkCore.Design и Microsoft.Extensions.Hosting.

После этого был настроен контейнер Dependency Injection в файле App.xaml.cs. В контейнер был зарегистрирован ApplicationContext с указанием строки подключения к базе данных SQL Server. Также были зарегистрированы ContactsListViewModel, ContactEditViewModel и главное окно приложения. Благодаря этому контекст базы данных передаётся во ViewModel через конструктор, а не создаётся вручную внутри классов.

На следующем этапе была реализована операция чтения данных. В ContactsListViewModel при создании ViewModel вызывается метод загрузки контактов из базы данных. Данные получаются из _context.Contacts, сортируются по идентификатору и помещаются в коллекцию ObservableCollection<Contact>, которая отображается в интерфейсе с помощью DataGrid. Дополнительно была реализована фильтрация контактов по имени и номеру телефона. Фильтрация выполняется по уже загруженной коллекции без дополнительного обращения к базе данных.

Затем была реализована операция создания нового контакта. Для этого была создана форма редактирования, в которую пользователь вводит имя и номер телефона. При сохранении создаётся новый объект Contact, после чего он добавляется в контекст методом _context.Contacts.Add(newContact). После вызова _context.SaveChanges() Entity Framework Core формирует SQL-команду INSERT, и новый контакт сохраняется в таблице базы данных. После успешного добавления список контактов обновляется в интерфейсе.

Также была реализована операция редактирования контакта. Пользователь выбирает контакт в таблице и нажимает кнопку изменения. Данные выбранного контакта передаются в форму редактирования. При сохранении приложение получает соответствующую запись из базы данных по идентификатору, изменяет свойства Name и Phone, после чего вызывает _context.SaveChanges(). Так как объект получен из текущего контекста, EF Core отслеживает его изменения и автоматически формирует команду UPDATE.

После этого была реализована операция удаления. Пользователь выбирает контакт из списка и нажимает кнопку удаления. Перед удалением выводится окно подтверждения. Если пользователь подтверждает действие, контакт находится в базе данных по идентификатору, передаётся в метод _context.Contacts.Remove(contactFromDb), после чего вызывается _context.SaveChanges(). В результате Entity Framework Core формирует SQL-команду DELETE, запись удаляется из таблицы, а интерфейс обновляется.

На заключительном этапе была добавлена обработка ошибок. Операции сохранения, изменения, удаления и загрузки данных были помещены в блоки try-catch. Это позволяет обработать возможные ошибки подключения, нарушения ограничений базы данных или другие исключения, возникающие при работе с SQL Server. После каждой успешной операции вызывается обновление списка контактов, благодаря чему пользовательский интерфейс всегда отображает актуальные данные из базы.

Результат выполненной работы

В результате выполнения лабораторной работы было модернизировано WPF-приложение «Телефонная книга» с использованием Entity Framework Core. В проекте Lab_13 был реализован полный набор CRUD-операций для работы с таблицей Contacts. Приложение позволяет загружать контакты из базы данных, выполнять поиск по имени и номеру телефона, добавлять новые записи, редактировать существующие контакты и удалять выбранные данные.

Проверка работы программы показала, что все операции выполняются напрямую через ApplicationContext без использования дополнительных паттернов абстракции. Добавление контакта выполняется через метод Add() и сохраняется в базе после вызова SaveChanges(). Редактирование реализовано через изменение свойств объекта, полученного из контекста, что позволяет использовать механизм Change Tracker. Удаление выполняется через метод Remove() с последующим сохранением изменений. После каждой операции интерфейс приложения обновляется и отображает актуальные данные.

В ходе лабораторной работы были закреплены практические навыки работы с классом DbContext, свойством DbSet, механизмом Change Tracker и жизненным циклом сущностей. Было изучено, как состояния Added, Modified, Deleted, Unchanged и Detached влияют на формирование SQL-команд при вызове SaveChanges(). Также были закреплены навыки внедрения ApplicationContext во ViewModel через Dependency Injection и обработки ошибок при работе с базой данных. Итоговое приложение стало полноценной телефонной книгой с поддержкой чтения, добавления, редактирования и удаления контактов в базе данных SQL Server.
